# 🏛️ التقرير المعماري والتشخيصي الشامل: مشروع (Django Fatawa API)
**تاريخ التقرير:** أبريل 2026
**الهدف:** توثيق الحالة الفنية للمشروع، ومسار التطوير، ليكون مرجعاً (State of the Union) لدمجه مع أي نموذج ذكاء اصطناعي قادم.

---

## 1️⃣ البنية التحتية والتقنيات المستخدمة المتوفرة فعلياً (Tech Stack)
النظام مبني على أحدث وأكثر التقنيات استقراراً للـ Backend، وهي خالية من التضارب (Conflict-Free):
*   **اللغة الأساسية:** Python 3.11 (داخل الحاويات).
*   **إطار العمل:** Django 4.2.7 مع (Django REST Framework 3.14).
*   **قاعدة البيانات:** PostgreSQL 15 (تدار بواسطة Docker Compose).
*   **نظام المصادقة:** JWT (عبر مكتبة djangorestframework-simplejwt 5.3).
*   **التوثيق والمواصفات (OpenAPI):** تعمل عبر `drf-spectacular` لإنتاج روابط الـ Swagger والـ Redoc.
*   **البيئة التشغيلية:** حاويات Docker بالكامل (`db`, `web`).

---

## 2️⃣ هندسة المكونات (حالة الملفات والمجلدات)
المجلد الجذري للمشروع مرتب، ويحتوي على تطبيق واحد رئيسي فعّال حتى اللحظة وهو `accounts`.

### 🛡️ تطبيق (Accounts) - "النواة الأمنية المكتملة"
تطبيق جبار ومعقد للغاية تم بناؤه من الصفر ويحتوي على قلب نظام الـ (RBAC - Role Based Access Control):
*   `models.py`: نموذج `User` مخصص يعتمد على `UUID`، مع جداول `Role` و `Permission` وعلاقات M2M ذكية وتتبع حالة `is_custom`.
*   `serializers.py`: مترجمات (Serializers) محصنة تمنع التكرارات (IntegrityErrors) وتحتوي على دوال التسجيل (Register) وتغيير كلمة المرور. (الاستجابات باللغة الإنجليزية).
*   `permission_sync.py`: ملف يحتوي على ميكانيكية استماع (Django Signals) متقدمة جداً، تلتقط أية تغييرات في أدوار المستخدمين وتدرجها في جدول صلاحياتهم بشكل فيزيائي. تم حل مشاكل الـ Circular Imports هنا.
*   `enhanced_views.py`: الـ Controllers للنظام! تم نقلها من `views.py`، وهي تحوي دوال تسجيل الدخول، معلومات الحساب `/api/auth/me/`، والـ Endpoints المحمية.
*   `permissions.py`: يحتوي على الحارس العملاق `HasAccess` الذي يحدد من يمكنه دخول أي Endpoint بناءً على المتغير `required_permission`.

## 3️⃣ تحليل المحادثات السابقة والقرارات المعمارية
طوال رحلتنا (كـ Pair Programmers)، تم التركيز على تصميم نظام حماية غير قابل للاختراق.
**المشاكل التي تم تدميرها في المحادثات السابقة:**
1.  **مشكلة 500 IntegrityError:** كان الـ API ينهار عند تسليم صلاحية موجودة مسبقاً لمستخدم. قمنا بمعالجة ذلك بذكاء داخل دوال الـ `validate()` في السريالايزر.
2.  **مشكلة المزامنة اليدوية للصلاحيات:** لوحة الأدمن والـ API كانا متباعدين، قمنا بتوحيدهما باستخدام نظام `post_save` و `post_delete` (Signals) بحيث أياً كانت طريقة إدخال الصلاحية (API أو لوحة تحكم) ستتزامن تلقائياً.
3.  **مشاكل اختبار الـ Hash:** أصلحنا نقطة إنشاء الحسابات بـ `set_password` وتأكدنا، وأجرينا اختبارات (`Integration Tests`) في ملف `run_e2e_tests.py`.
4.  **منع الردود العربية في الـ API:** تم تنظيف كافة ردود الـ API (`Responses`) لتكون باللغة الإنجليزية القياسية (`"Password changed successfully"`).

## 4️⃣ استنتاجات: نقاط القوة 🟢 ونقاط الضعف 🔴

### 🟢 نقاط القوة (Strengths)
*   **عزل الهوية (Identity Isolation):** نظام التحكم المخصص الذي صنعناه يسهل ربطه بأي تطبيق مستقبلي مهما بلغت مساحته وحجمه، لأنه يدعم الـ (Micro-Permissions).
*   **جاهزية التوليد (Production Readiness):** كل Endpoints النظام تتوافق مع معايير OpenAPI ويتم اصطياد الأخطاء بأناقة تامة لا تضرب الـ Server.
*   **وجود Postman Collections ممتازة.**

### 🔴 نقاط تحتاج لتدخل سريع (Weaknesses & Fixes required)
*   **غياب الـ Volume Binding في الـ Docker:** في ملف `docker-compose.yml` الحالي، خادم الـ `web` يعمل بنسخ الملفات `COPY . /app/` دون ربط مباشر (`volumes`). هذا ما أجبرنا على تدمير الحاوية وإعادة بنائها مع كل تغيير بسيط! **يجب حل هذا أولاً.**
*   **انعدام تطبيق الأعمال الاساسي (Business Logic):** النظام عبارة عن حماية وهياكل بدون جوهر، فلا يُوجد حتى الآن تطبيق الفتاوى المنشود (`fatawa`).

## 5️⃣ خطة الطريق للمستقبل (للذكاء الاصطناعي والمهندس القادم)
السطور التالية تعتبر أوامر ملزمة للمرحلة القادمة:

1.  **المهمة الأولى:** إضافة `volumes: - .:/app` وتفعيل الـ `--reload` في الـ `command` الخاص بالـ Web في ملف `docker-compose.yml` لجعل التطوير سريعاً.
2.  **المهمة الثانية:** إنشاء تطبيق `django-admin startapp fatawa` وإضافته في ملفات الإعدادات.
3.  **المهمة الثالثة:** بناء نماذج الفتاوى (Models) والتي تشمل `Question`, `Answer`, `Category` واستخدام `UUID` أيضاً.
4.  **المهمة الرابعة:** جلب حارسنا `HasAccess` من `accounts.permissions` وربطه مع الـ `FatawaViewSet` لحماية المسارات.
